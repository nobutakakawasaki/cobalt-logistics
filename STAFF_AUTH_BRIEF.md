# 社員登録・ログイン + 活動ログ ブリーフ #8

管理画面(wp-admin)は既にログイン必須（WordPress標準機能）だが、現状は`admin`ユーザー1つ
のみ。社員ごとにアカウントを分け、社員ID＋パスワードで登録・ログインできるようにし、
「誰が・いつ・何を編集/ログインしたか」を可視化する。

日本の企業システム（freee人事労務、ジンジャー人事労務、マネーフォワードクラウド等）の
一般的な構成を調査した結果、ログイン画面は「ログインID＋パスワード」、登録画面は
「社員コード・氏名・メールアドレス・パスワード」が標準的な項目だった。これに準拠する。

## セキュリティ方針（最重要）

- **認証処理は独自実装せず、WordPressコア機能のみを使う**: ユーザー作成は`wp_insert_user()`、
  パスワードハッシュ化はWordPressコアが自動で行う（`wp_hash_password()`を直接呼ぶ必要はない、
  `wp_insert_user()`に平文パスワードを渡せば内部で安全に処理される）。ログイン処理は
  `wp_signon()`を使う。独自のパスワードハッシュ・セッション管理は絶対に実装しないこと。
- **登録は完全公開にしない**: 社員登録フォームには「登録コード」欄を設け、
  `cobalt_logistics_staff_registration_code`というオプション値（デフォルト値を
  functions.phpに定数として持たせ、コメントで「実運用時は変更すること」と明記）と
  一致しない場合は登録を拒否する。これにより、URLを知っているだけの第三者が
  勝手に編集権限アカウントを作れないようにする。
- **新規登録ユーザーのロールは `editor`**（記事の作成・編集ができるが、
  ユーザー管理やテーマ編集などの管理者権限は持たない）。
- 全てのフォームに `wp_nonce_field()` / `check_admin_referer()` (または
  `wp_verify_nonce()`) を必須で使う。
- ログイン失敗時のエラーメッセージは「社員IDまたはパスワードが正しくありません」の
  ように、どちらが間違っているか特定できない汎用文言にする（アカウント存在の
  列挙攻撃を防ぐ、既存の問い合わせフォームのバリデーションとは別の配慮点）。
- 入力値は全て `sanitize_text_field()` 等でサニタイズし、出力時は `esc_html()`/`esc_attr()`
  を徹底する（既存の問い合わせフォーム実装と同じ水準）。

## 1. 社員登録ページ

新規ページテンプレート `page-staff-register.php`（Template Name: 社員登録）を作成し、
`/staff-register/` に配置する。**メインナビゲーション・フッターには追加しない**
（一般公開の営業用ページではなく社内向けのため。URLを知っている人だけがアクセスする想定）。

フォーム項目:
- 氏名（必須）
- 社員ID（必須、半角英数字、一意である必要がある。既に使われている場合はエラー）
- メールアドレス（必須、`is_email()`で検証、既存ユーザーと重複時はエラー）
- パスワード（必須、8文字以上を要件にする）
- パスワード（確認用、一致チェック）
- 登録コード（必須、上記のオプション値と一致するかチェック）

処理: `admin-post.php` 経由（既存の `cobalt_submit_inquiry` と同じ仕組み）で
`cobalt_staff_register` アクションを新設し、バリデーション通過後 `wp_insert_user()`
でユーザー作成（`user_login`は社員IDそのもの、`role`は`editor`、`user_email`を設定）。
社員IDは追加でユーザーメタ `cobalt_staff_id` にも保存する（`user_login`と同じ値だが、
将来ログインIDの命名規則を変える場合に備えて独立させておく）。

成功時は登録完了メッセージ＋ログインページへのリンクを表示する
（実際のメール送信は行わない、デモのため）。

## 2. 社員ログインページ

新規ページテンプレート `page-staff-login.php`（Template Name: 社員ログイン）を
`/staff-login/` に配置する。ナビゲーションには追加しない。

フォーム項目: 社員ID、パスワード。

処理: WordPressの `authenticate` フィルターに独自関数をフックし、入力が社員ID形式
（`cobalt_staff_id` メタで一致するユーザーが見つかった場合）であれば、対応する
ユーザーの `user_login` に変換してから通常の認証処理に渡す（既存のユーザー名/
メールアドレスでのログインも壊さないよう、フィルターは「他の認証方法が失敗した場合の
追加手段」として実装し、既存のWordPress認証ロジックを置き換えないこと）。
ログインフォーム自体は `wp_signon()` を使い、成功時は `/wp-admin/` へリダイレクトする
（実際のコンテンツ編集はwp-admin側で行うため、フロント側に独自の編集画面は作らない）。

## 3. 活動ログ（誰が・いつ・何をしたか）

新しい非公開カスタム投稿タイプ `activity_log` を追加する（既存の `inquiry` CPTと
同じパターン: `public => false`, `show_ui => true`, 管理画面の内部データ専用）。

以下の2つのフックでログを記録する:
- `wp_login` アクション: ログイン成功時に1件記録（タイトル: "ログイン: {表示名}"、
  メタ: `log_type=login`, `log_user_id`）
- `save_post` アクション（`wp_is_post_autosave()` / `wp_is_post_revision()` の
  場合は除外し、実際の保存時のみ記録。対象は `job`・`post`（news/column）の
  編集のみでよく、`activity_log` 自身や `inquiry` の保存はログ対象外にすること
  ── 無限ループ・ノイズ防止）: 1件記録（タイトル: "編集: {投稿タイトル}"、
  メタ: `log_type=edit`, `log_user_id`, `log_post_id`）

管理画面ダッシュボードに新しいウィジェット（`wp_dashboard_setup` フックで
`wp_add_dashboard_widget()`）を追加し、直近20件の活動ログを新しい順に一覧表示する
（誰が／いつ／何をしたか、が一目でわかるテーブル形式でよい）。

## 制約

- 既存の全機能（問い合わせフォーム・地図・カーソル追従・ルートライン・シミュレーター・
  スクロールリビール・導入事例・サービス画像・追従CTAバー・ヒーロースライドショー・
  社員の声・選考の流れ・求人詳細ページ・お知らせ・コラム）を壊さないこと。
- `job` CPT・`inquiry` CPT・その自動シード関数（`cobalt_logistics_seed_jobs`、
  `cobalt_logistics_seed_news_column_*`）には触れないこと。
- vanilla PHP/CSSのみ、外部認証ライブラリは使わない。
- レスポンシブ対応必須。

## 完了後の確認

- `php -l` 全ファイル、`docker compose logs wordpress --tail=50` でエラーが無いこと
- `/staff-register/` `/staff-login/` を含む既存23URL + 2URL、計25URL全て200
- 実際に登録コードなしで登録を試み、拒否されることを確認する
- 実際に正しい登録コードで社員登録を行い、作成されたユーザーのロールが `editor`
  であることを `wp user list` で確認する（テスト後、作成したテストユーザーは削除すること）
- 作成したユーザーで実際にログインが成功することを確認する（社員IDでのログイン）
- 誤ったパスワードでログインを試み、エラーメッセージがアカウントの存在有無を
  特定できない汎用文言になっていることを確認する
- 活動ログにログイン・編集操作が正しく記録され、ダッシュボードウィジェットに
  表示されることを確認する
- 問い合わせフォーム・求人機能・お知らせ/コラムに回帰がないこと（再送信テストで確認）
